## 4. "OFFERS" — NEW NAV PAGE & FULL PROMOTIONS ENGINE

### 4A. Navigation

Create a new page file: `client/src/pages/offers.tsx`.

### 4B. Offers Data Model
Create a new Mongoose model `server/models/Offer.ts`:

```typescript
{
  name: String,           // e.g. "Christmas Sale 2024", "Summer Deals"
  description: String,
  isActive: Boolean,      // Toggle on/off without deleting
  startDate: Date,
  endDate: Date,
  offerType: "percentage_discount" | "b1t1" | "buy1_take_percentage" | "flat_discount",
  // For percentage_discount: item-level discount percentages (not global)
  // For b1t1: buy 1 get 1 free of same item
  // For buy1_take_percentage: buy 1, get second at X% off
  // For flat_discount: flat ₱ amount off per item
  items: [
    {
      itemId: ObjectId (ref Item),
      itemName: String,       // denormalized for display
      discountValue: Number,  // percentage (0-100) or flat amount in ₱
      // For b1t1, discountValue is ignored (always 100% off second)
      // For buy1_take_percentage, discountValue = % off the second item
    }
  ],
  createdBy: ObjectId (ref User),
  createdAt: Date,
  updatedAt: Date
}
```

Add to `shared/schema.ts`: `createOfferSchema`, `updateOfferSchema`, `OfferItem` type.

### 4C. Offers Page UI (`client/src/pages/offers.tsx`)

### 4D. Applying Offers to Orders
In the backend `POST /api/orders` and the order schema:
- When creating an order, check for active offers on each item.
- For each item in the order, find any active offer covering that item.
- Apply the discount:
  - `percentage_discount`: `lineTotal = qty * price * (1 - discount/100)`
  - `b1t1`: if qty >= 2, customer pays for ceil(qty/2) items (buy 1 take 1, buy 3 take 1 extra, etc.)
  - `buy1_take_percentage`: the second item in each pair costs `price * (1 - discount/100)`
  - `flat_discount`: `lineTotal = qty * (price - flatAmount)`, min ₱0
- Store on the order item: `{ itemId, itemName, qty, originalUnitPrice, discountedUnitPrice, discountApplied: boolean, offerName: string, lineTotal }`
- Do NOT deduct stock for the "free" items in B1T1 (they are still physical items leaving inventory — DO deduct them but flag them).

### 4E. Offers API Routes (add to `server/routes.ts`)
GET    /api/offers              → list all offers (with pagination + filter by active/inactive)
POST   /api/offers              → create new offer
PUT    /api/offers/:id          → update offer
DELETE /api/offers/:id          → delete offer
PATCH  /api/offers/:id/toggle   → toggle isActive
GET    /api/offers/active       → list currently active offers (for use in Create Order)

## 9. CODING STANDARDS TO FOLLOW

- All new API routes must have Zod validation using `z.parse()` or `z.safeParse()` before touching the DB.
- All new frontend data fetching uses TanStack Query (`useQuery` / `useMutation`).
- Invalidate relevant query caches after mutations (e.g., after creating an offer, invalidate `["/api/offers"]` and `["/api/dashboard/stats"]`).
- Use shadcn/ui components for all UI: Dialog, Sheet, Select, Combobox (Command component), Badge, Accordion, Switch, Tabs.
- Use `cn()` from `lib/utils.ts` for all conditional class merging.
- Define all new TypeScript types from the Zod schemas using `z.infer<typeof schema>`.
- Follow the existing `apiRequest()` pattern for all fetch calls.
- Every user-facing action that succeeds should show a `toast({ title: "...", description: "..." })`.
- Every error from the API should show a `toast({ title: "Error", description: err.message, variant: "destructive" })`.
- Keep all color coding consistent: amber = pending/warning, green = success/paid, red = error/cancelled, blue = processing/online, purple = reservation.
- Do not remove or break any existing functionality. All existing routes, pages, and features must continue to work.
- After all changes, run `npm run check` (TypeScript check) and fix all type errors before finishing.
